## 前言

Obsidian Projects 插件是一款项目管理插件，开发者是现在 Obsidian 官方教程的主要创作者 Marcus Olsson ，他对于为什么构建这个插件有以下的想法：

**来自 @pseudometa 的提问**：看起来它是一个很强大的插件，您可以详细说明一下它的用途或者计划吗？例如，如果您只是偶尔使用 Kanban 插件，它可以替代掉它的作用吗？还是说它是为了和这些插件一起使用准备的？

> **来自作者的回答是**：我并不打算用它来取代掉任何插件。我的目标是将它们结合起来。最初我建立 Projects 的动机是，现在在社区中已经有这么多的插件可以以不同的方式显示你的笔记中的数据，例如 Kanban、Fantasy Calendar 等等。问题是，它们都会倾向于使用不同的数据源和配置格式。有时你甚至需要为该特定插件添加自定义属性。-
> 而现在我开发了 Projects，我想要在插件里面将这个现状的问题给解决一下。简而言之，Projects 插件可以从文件夹中的笔记中获取结构化的数据，或者从 Dataview 索引数据。然后你可以选择使用不同的内置视图以不同的方式显示数据 —— Projects 会为每个视图提供独立配置以及数据源。-
> Projects 默认附带三个视图，我的想法是——其他插件可以将自身注册为一个 Projects 的视图或者给 Projects 提供一个视图。例如，Projects 插件给 Kanban 插件一个确定的数据结构，其中包含正在进行的任务的数据，而 Kanban 插件则利用他们渲染对应的 HTML 元素。-
> 不过更进一步地说，我希望 Projects 插件可以提供大量可复用的插件代码来帮助开发者来减轻对应的工作。同时，我现在还在研究怎么提供更好的视图相关的 API，而如果我认为 API 已经能让我满意的话，我可能会开始向其他插件作者发送 PR 来将他们整合进来。

我的想法：

Obsidian Projects 的特点在于它是开箱可用的，也即你只需要打开它，然后打开一个文件夹，它就可以完整地显示对应地文件夹中的每个文件的元数据。

以下会根据它目前 0.18.0 版本的功能来介绍一下它的使用方法和使用场景。

## 功能介绍

Obsidian Projects 的使用步骤相当简单，在安装后你只需要在左侧栏点击 Projects 的图标、在命令中输出 Show Projects 就可以打开对应的窗口。或者你也可以直接在文件夹上右键点击 `Create project in folder` 来创建新的 Project。

初次打开 Projects 现在应该已经提示你可以选择一个文件夹，或者尝试使用 Projects 内置的 Demo Projects，打开后可以看到以下的界面

![](https://image.cubox.pro/article/2022101909015230077/97813.jpg)

1、你可以选择 1 来直接新建一个 Project ，注意，你可以同时拥有多个 Project，也即你可以多个 Project 同时在 Projects 中管理；-
2、你可以选择查看插件提供的 Demo 库来了解这个插件的基本运行逻辑；-
3，4，5、都是不同的打开 Projects 的方式。

### Project 配置窗口

![](https://image.cubox.pro/article/2022101909015278720/95251.jpg)

在 Obsidian 插件窗口中你可以看到以下的一些配置项：

1、你的 Project 命名，会显示在你选择不同的 Project 时候的下拉列表中；-
2、使用 Dataview 来索引数据，当你没有安装 Dataview 的时候会默认不显示；-
3、使用文件夹地址来索引数据，当留空的时候是索引根目录（也即理解成可以索引全库）；-
4、包括子路径，如果不包括的话，仅索引指定文件夹的内容，而不会索引指定文件夹下的子文件夹内容；-
5、默认的笔记名；-
6、模板，你可以设置多个模板；

全都配置好后点创建 Project 就可以进入到 Projects 的窗口了。

### Projects 操作窗口

> 表格视图

![](https://image.cubox.pro/article/2022101909015270101/43797.jpg)

1、选择不同的 Project ；-
2、编辑或者删除当前的 Project ；-
3、新建按钮，用于选择新建 Project、视图或者笔记；-
4、隐藏表格中的列 5、选择不同的视图，点击既可切换；-
6、Tag 列，当你在索引的笔记中任一 YAML 中填入 Tag 字段的时候，你就可以在这个表格中看到对应的列，而 Tag 列是相对特殊的，你可以在这个单元格中添加或者删除标签；-
7、Number 列，你可以填入数字，当对应的 YAML 属性是 number 类型的时候，这里面就只能填入数字了；-
8、Boolean 列，当你设置，例如 published: true 或者 false 的时候就可以直接在表格中点击切换；-
9、日期列，当你的 YAML 数据是 YYYY-MM-DD 或者 YYYY/MM/DD 的时候，这里就会显示成日期的属性，你点选的时候可以直接选择新的日期；-
10、文本列，你在这里填入你的文本内容；-
11、添加新的笔记

有些情况下你可能只需要一个表格就足够了，不过还漏了一点，你要跳转到**对应的笔记的话，需要点击行的左侧的数字**。

* * *

> 日历视图

![](https://image.cubox.pro/article/2022101909015267822/61659.jpg)

1、你点击这里可以添加笔记；-
2、点击事件，可以直接修改事件的元数据；-
3、选择基于什么日期来排序事件；-
4、选择基于什么属性来确定已经完成；-
5、你可以选择：日、月、周、双周视图来看所有的事件

以上虽然简单，但是在实际使用过程中这个使用起来非常的方便，方便的原因是你可以很方便地查看对应的元信息，配合 Full Calendar 会很方便。

* * *

> 看板视图

![](https://image.cubox.pro/article/2022101909015238720/12221.jpg)

1、添加新的笔记，而且附带对应的状态作为 YAML ；-
2、优先级，根据 4 来调整变化；-
3、调整状态，Kanban 根据这个来生成不同的列；-
4、调整权重，Kanban 根据这个来排序不同的卡片。

点击任意的卡片也可以正常跳转到对应的笔记中去。

## 应用场景

在 Projects 之前，我是用 Kanban 来管理项目的，但是 Kanban 只提供了单个界面，也即无法查看对应的日历，也无法提前安排相关项目等；但是利用 Projects 你相当于拥有了基于文件夹的数据库，你可以将一个文件夹视作数据库（这个和 Dataview 这种插件不太一样的地方是，Projects 允许你直接编辑对应的元数据）来操作。甚至和 Database Folder 也存在比较大的区别，因为 Database Folder 也仅限于拿来作为数据表进行操作，而且需要额外在你的文件中加入一个文件来标识对应的项目。

所以考虑到项目相关，那就有以下的场景：

### 写作

Obsidian 自 0.6.0 版本就一直是我的主力的文档编辑器（不考虑 Word 的话），而如何规划每篇教程或者分享的写作一直是我比较头痛的问题。我尝试过使用外部的任务管理软件，Obsidian 内部的 Tasks、Cardboard、Kanban、Dataview 以及 DB Folder 插件来管理自己的写作流程，但是都无一例外坚持不下去，因为每次一旦我想要编辑内容的时候，如果是外部的任务管理软件，我需要回到 Obsidian 找到对应的文档；而如果是内部的任务管理插件，那我就无法实时编辑对应的元数据，也缺少了对应的视图来查看任务内容。

所以支持日历+数据视图的 Projects 就能够在我的工作需求下起到比较好的帮助了。我的工作流可以简单规划成以下的几步：

*   构思下一篇内容需要写什么主题
*   确定了主题以后新建一个文档在 Obsidian 中
*   在 Projects 中查看自己接下来的一段时间内需要做的事情
*   然后排期以及安排各个需要写的内容的时间
*   等待完成

以上的工作流程可以自由配合 Dataview / Kanban / Full Calendar 使用。

### 工作

对于我来说，Projects 在工作上的应用更多，凸显在以下的方面（道）：

*   单个项目可以控制，因为我工作中的每个项目就等于一个文档，所以写文档作为项目控制是我最原始的诉求；
*   简单而有效的时间和主题追踪，可以和 Dataview 配合索引
*   多个项目同时关注

第一点是受到我本身的工作限制的必然结果；第二点则是考虑到笔记库和文档库的数据的控制使用；第三点则是我同时处理的东西确实不少。

而第三点相比原先的 Kanban 的方案又有一个特别点在于——它可以一个界面查看所有的 Projects，而所有有 Kanban 的笔记则需要额外的查询方式才能聚堆所有 Kanban，也即又需要额外增加新的一个未来不可用的文档。

第二点则是 Obsidian 原生的纯文本和插件的互相作用的特点，Kanban、Full Calendar 甚至 Memos 都可以互相配合起来，进而保证数据之间的互通互联。

更详细的工作步骤：

*   我针对每个客户/服务对象/插件项目构建了一个独立的文件夹，用于监控每个客户/服务对象/插件项目的动态；
*   每个客户/服务对象/插件项目的名下有对应的交互记录，由于单次的交互都是固定的，所以我新建笔记来记录单次的交互；
*   在第一步的基础上，我构建了全局的客户/服务对象/插件项目的文件夹，而且索引了所有的子文件夹，进而在一个全局的角度上观察我近期在跟进的客户或者项目或者插件；
*   在上述的基础上，进一步地，我将每个客户/服务对象/插件项目的项目进行互相的排期，接着，在全局的角度上查看每天的事项以及对应的优先级——这么说可能有点抽象，具体来说就是每个客户/服务对象/插件项目都有不同的处理排期，在上述的全局的角度下可以看到哪些天的排期过多，可以提前安排对应的任务项。
*   为了保证渐进地记录客户/服务对象/插件项目的情况，会利用 Hover Editor 来实时编辑对应的内容和文档；同时利用 Dataview 来索引所有的内部任务（分成多个笔记不同的需求链）
*   在每周或者每月的总结上，截取已经完成的部分，再次利用 Dataview 进行快速的筛选生成表格后粘贴到 Word 里面去。

一些细节：

*   还要解决上述的每个客户/服务对象/插件项目的对应项目的快速处理， Projects 中提供了创建模板笔记的方案，利用 Templater 的监控目录功能，更快地实现了大部分的模板化文档的构建需求。

## 总结

以上就是 Projects 插件的简单介绍，你说它现在是否已经有 Notion Database 的特性和功能，那显然还是没有的，但是它确实解决了在 Obsidian 中进行初步的项目管理的需求，尤其对我这种非重度的项目管理需求的用户来说十分有用。
